# AGENTS.md

本文件定义本项目中 AI 代理协作与开发工作的基础规范。

## 1. 代码组织结构：路由与索引

### 1.1 通用原则

代码必须保持结构清晰，禁止把所有逻辑堆到一个文件里。

当文件出现以下信号时，必须主动评估是否需要拆分，而不是继续往里堆代码：

- 行数过长。
- 职责混杂。
- 存在难以复用的重复代码。
- 单个文件需要同时承载多个明显不同的变化原因。
- 阅读、测试或修改该文件时需要同时理解过多上下文。

### 1.2 技术栈到 skill 的路由表

在写代码、重构代码、评估代码组织方式或判断是否需要拆分文件之前，必须先根据当前涉及的语言、框架或平台，加载对应的 skill。

具体拆分标准、目录习惯、文件边界和测试组织方式，以对应 skill 内容为准。不要凭聊天记忆、训练经验或主观偏好随意判断。

| 技术栈 / 语言 / 框架 | 对应 skill 文件 | 状态 |
| --- | --- | --- |
| Rust / Cargo | `.agent/skills/rust-code-organization/SKILL.md` | 已启用 |
| Go / Go Modules | `.agent/skills/go-code-organization/SKILL.md` | 已启用 |
| Tauri / React / TypeScript / Vite | `.agent/skills/tauri-react-typescript-vite/SKILL.md` | 已启用 |

项目级 skills 放置在 `.agent/skills/` 目录下。

当 Rust 任务涉及新增或修改测试时，还必须加载 `.agent/skills/rust-testing/SKILL.md`。

当 Rust 任务涉及错误类型、public error API 或 `Result<T, E>` 设计时，还必须加载 `.agent/skills/rust-error-design/SKILL.md`。

当 Go 任务涉及新增、修改或迁移测试，选择同包或外部测试 package、表驱动测试、集成测试、race 或 fuzz 策略时，还必须加载 `.agent/skills/go-testing/SKILL.md`。

当 Go 任务涉及错误类型、sentinel error、public error API、错误包装、`errors.Is` / `errors.As` 或跨 package 错误边界时，还必须加载 `.agent/skills/go-error-design/SKILL.md`。

当 Tauri 后端涉及 `src-tauri/` 下 Rust 模块拆分、Rust 测试或 Rust 错误 API 时，必须在 Tauri / React / TypeScript / Vite skill 之外，继续按上述 Rust 规则加载对应 Rust skill。

当前端或 Tauri 任务涉及测试时，必须加载 `.agent/skills/tauri-react-typescript-vite/references/testing.md`；如果同时涉及 Rust 测试，还必须加载 `.agent/skills/rust-testing/SKILL.md`。

### 1.3 路由表未覆盖时的处理

如果当前涉及的技术栈不在路由表中，必须：

1. 明确指出：当前缺少对应 skill。
2. 基于该语言或框架社区的通用最佳实践给出临时建议。
3. 明确说明这些建议是临时判断，不是本项目已确认规范。
4. 提醒需要补充对应技术栈 skill。
5. 在得到确认前，不得把临时建议当作项目固定规范写入代码结构决策中。

## 2. 调查与分析必须基于实证

任何情况下，分析项目、排查问题、评估实现、给出结论或提出方案，都必须从当前代码的具体实现出发。

代理必须先收集本地证据，再下结论。可使用的证据包括：

- 阅读实际代码、配置、文档、测试和脚本。
- 运行实际命令。
- 查看实际输出、错误信息、日志和测试结果。
- 检查当前目录结构、依赖声明和已有约定。

禁止依赖聊天记忆中的信息或未经验证的主观假设作为判断依据。

如果某个信息无法通过当前可获取的证据验证，必须明确说明：

> 这是假设，尚未验证。

未经验证的信息不能当作既定事实陈述，也不能作为关键决策依据。

## 3. 功能完工标准：测试驱动

实现功能时，必须考虑该功能在不同场景下是否都能满足需求，包括：

- 正常输入。
- 边界情况。
- 异常输入。
- 与现有行为的交互。
- 需求中明确或隐含的失败场景。

只有当所有相关测试都编写完成并通过之后，才能认为该功能已经完工。

不允许仅凭“代码写完了”就宣称任务完成。

每个测试用例必须包含描述性注释，清晰说明：

1. 测试目标：验证什么。
2. 构造方法：如何搭建测试场景，步骤化说明。
3. 输入数据：具体测试输入是什么。
4. 预期行为：期望的输出或结果是什么。

## 4. 代理协作模式

接收任务的代理，即主代理，必须充当协调者和最终审查者，而不是包揽所有具体执行工作。

非平凡的工作必须拆分为有明确边界的子任务，并分配给子代理执行。非平凡工作包括：

- 有一定复杂度的任务。
- 涉及多个步骤的任务。
- 涉及多个模块的任务。
- 需要研究、实现、审查、测试等多个环节的任务。

对子代理要有耐心。不要因为一两次不理想的结果就放弃拆分方式，除非已经证明该子代理确实无法胜任对应任务。

在可行的情况下，以下环节应当相互分离：

- 研究：了解现状、收集信息。
- 实现：编写或修改代码。
- 审查：检查质量、风险和一致性。
- 测试：验证功能和回归风险。

不要用同一个执行过程混杂完成所有环节，以保证每个环节都有独立、聚焦的关注点。

## 5. 需求澄清机制

每个需求在开始开发之前，必须确保没有疑问。

如果需求描述存在以下情况，必须先指出问题并与用户沟通确认，不能凭猜测自行决定实现方向：

- 描述存在歧义。
- 信息不完整。
- 与当前代码的实际实现存在冲突。
- 根据当前代码判断，按字面需求去做会导致结果和预期不符。
- 存在多个合理实现方向，且选择会影响用户体验、接口、数据结构、兼容性或后续维护。

沟通时必须具体说明：

1. 疑问点是什么。
2. 为什么会有这个疑问。
3. 如果疑问来自现有实现，说明它和现有哪部分代码或行为冲突。
4. 有哪些可能的实现方向供用户选择。
